# 쿠버네티스

> 컨테이너화된 애플리케이션의 배포, 확장, 관리를 자동화하는 오픈소스 컨테이너 오케스트레이션 플랫폼입니다.

---

## 1. 정의

쿠버네티스(Kubernetes, K8s)는 Google이 내부에서 사용하던 컨테이너 관리 시스템 Borg를 기반으로 2014년에 오픈소스로 공개한 [[GLOSSARY#컨테이너 오케스트레이션|컨테이너 오케스트레이션]] 플랫폼입니다. 현재는 CNCF(Cloud Native Computing Foundation)에서 관리하고 있으며, 컨테이너 오케스트레이션의 사실상 표준으로 자리잡았습니다.

[[Docker]]로 패키징한 컨테이너들을 대규모로 배포하고, 자동 확장하며, 장애 시 자가 복구하는 기능을 제공합니다.

---

## 2. 등장 배경 및 필요성

- **대규모 컨테이너 관리의 한계**: Docker는 단일 호스트에서 컨테이너를 실행하기에는 적합하나, 수십~수천 개의 컨테이너를 여러 서버에 배포하고 관리하려면 별도의 오케스트레이션이 필요합니다.

- **수작업 운영의 비효율**: 배포, 스케일링, 로드 밸런싱, 장애 복구 등을 수동으로 관리하면 운영 부담이 급격히 증가합니다.

- **멀티 클라우드 및 하이브리드 환경**: 특정 클라우드에 종속되지 않고 온프레미스, 퍼블릭 클라우드, 하이브리드 환경에서 일관된 방식으로 컨테이너를 운영할 수 있는 표준 플랫폼이 필요했습니다.

- **선언적 인프라 관리**: 원하는 상태를 선언하면 시스템이 자동으로 해당 상태를 유지하는 방식이 요구되었습니다.

---

## 3. 작동 원리 / 핵심 개념

### 3.1 클러스터 아키텍처

쿠버네티스 클러스터는 **컨트롤 플레인**과 **워커 노드**로 구성됩니다.

```
┌─────────────────────────────────────────┐
│            컨트롤 플레인                   │
│  ┌───────────┐  ┌──────────────────┐    │
│  │ API Server│  │ Controller Manager│   │
│  └───────────┘  └──────────────────┘    │
│  ┌───────────┐  ┌──────────────────┐    │
│  │   etcd    │  │    Scheduler     │    │
│  └───────────┘  └──────────────────┘    │
└─────────────────────────────────────────┘
          │
          ▼
┌──────────────┐  ┌──────────────┐
│  워커 노드 1   │  │  워커 노드 2   │
│ ┌──────────┐ │  │ ┌──────────┐ │
│ │  kubelet │ │  │ │  kubelet │ │
│ │kube-proxy│ │  │ │kube-proxy│ │
│ │ 컨테이너   │ │  │ │ 컨테이너   │ │
│ │  런타임   │ │  │ │  런타임   │ │
│ └──────────┘ │  │ └──────────┘ │
└──────────────┘  └──────────────┘
```

### 3.2 컨트롤 플레인 구성 요소

| 구성 요소 | 역할 |
|-----------|------|
| **API Server** | 클러스터의 중앙 진입점. 모든 구성 요소가 이 API를 통해 통신합니다. |
| **etcd** | 클러스터의 모든 상태 데이터를 저장하는 분산 키-값 저장소입니다. |
| **Scheduler** | 새로 생성된 파드를 리소스 상황에 따라 적절한 노드에 배치합니다. |
| **Controller Manager** | 레플리카 수 유지, 노드 상태 감시 등 클러스터 상태를 원하는 상태로 수렴시키는 컨트롤러들의 집합입니다. |

### 3.3 워커 노드 구성 요소

| 구성 요소 | 역할 |
|-----------|------|
| **kubelet** | 각 노드에서 실행되며, 파드 내 컨테이너가 정상 동작하도록 관리합니다. |
| **kube-proxy** | 노드의 네트워크 규칙을 관리하고, 파드 간 통신 및 서비스 기반 로드 밸런싱을 처리합니다. |
| **컨테이너 런타임** | 실제로 컨테이너를 실행하는 소프트웨어입니다. containerd, CRI-O 등이 사용됩니다. |

### 3.4 핵심 워크로드 오브젝트

**파드(Pod)**

쿠버네티스에서 배포 가능한 가장 작은 단위입니다. 하나 이상의 컨테이너를 포함하며, 같은 파드 내 컨테이너들은 네트워크와 스토리지를 공유합니다.

**디플로이먼트(Deployment)**

파드의 선언적 업데이트를 관리합니다. 원하는 레플리카 수를 명시하면 해당 수만큼 파드를 유지하고, 롤링 업데이트와 롤백을 지원합니다.

**서비스(Service)**

파드 집합에 대한 안정적인 네트워크 엔드포인트를 제공합니다. 파드는 생성/삭제될 때마다 IP가 변경되므로, 서비스가 고정된 접근점을 제공합니다.

| 서비스 유형 | 설명 |
|-------------|------|
| **ClusterIP** | 클러스터 내부에서만 접근 가능한 기본 유형 |
| **NodePort** | 각 노드의 특정 포트로 외부 접근 허용 |
| **LoadBalancer** | 클라우드 로드 밸런서를 프로비저닝하여 외부 트래픽 수신 |

**네임스페이스(Namespace)**

하나의 클러스터를 논리적으로 분리하는 가상 클러스터입니다. 팀, 환경(dev/staging/prod), 프로젝트별로 리소스를 격리할 수 있습니다.

### 3.5 선언적 관리 방식

쿠버네티스는 원하는 상태를 YAML 매니페스트로 선언하면, 컨트롤러가 현재 상태를 원하는 상태와 지속적으로 비교하여 자동으로 맞춥니다.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
      - name: web
        image: nginx:1.25
        ports:
        - containerPort: 80
```

위와 같이 레플리카 3을 선언하면, 파드가 장애로 종료되더라도 쿠버네티스가 자동으로 새 파드를 생성하여 3개를 유지합니다.

## 4. 장점 및 이점

- **자가 복구(Self-healing)**: 장애가 발생한 컨테이너를 자동으로 재시작하고, 응답하지 않는 파드를 교체하며, 헬스 체크에 실패한 컨테이너의 트래픽을 차단합니다.

- **수평적 자동 확장**: CPU, 메모리 사용률 또는 커스텀 메트릭 기반으로 파드 수를 자동 조절하는 HPA(Horizontal Pod Autoscaler)를 제공합니다.

- **서비스 디스커버리와 로드 밸런싱**: DNS 이름 또는 IP 주소를 통해 파드를 자동으로 발견하고, 트래픽을 분산합니다.

- **롤링 업데이트와 롤백**: 무중단 배포를 기본 지원하며, 문제 발생 시 이전 버전으로 즉시 롤백할 수 있습니다.

- **클라우드 독립성**: AWS, GCP, Azure 등 주요 클라우드 및 온프레미스 환경에서 동일한 방식으로 운영할 수 있어 벤더 종속을 방지합니다.

- **풍부한 생태계**: Helm(패키지 매니저), Istio(서비스 메시), Prometheus(모니터링), ArgoCD(GitOps) 등 방대한 도구 생태계를 갖추고 있습니다.

---

## 5. 한계점 및 고려사항

- **높은 학습 곡선**: 파드, 서비스, 인그레스, 컨피그맵, 시크릿 등 다양한 오브젝트와 개념을 이해해야 하며, YAML 매니페스트 작성에 익숙해져야 합니다.

- **운영 복잡성**: 컨트롤 플레인 관리, 네트워크 정책, RBAC(역할 기반 접근 제어), 스토리지 관리 등 운영 부담이 큽니다. 관리형 서비스(EKS, GKE, AKS)를 사용하면 일부 부담을 줄일 수 있습니다.

- **소규모 환경에서의 과잉**: 소수의 컨테이너만 운영하는 환경에서는 [[ECS]]와 같은 단순한 오케스트레이션 도구가 더 적합할 수 있습니다.

- **리소스 오버헤드**: 컨트롤 플레인 자체가 CPU, 메모리를 소모하므로, 소규모 클러스터에서는 전체 리소스 대비 오버헤드 비율이 높아집니다.

## 6. 실무 적용 가이드

### 6.1 관리형 쿠버네티스 서비스 비교

직접 클러스터를 구축하는 대신 클라우드 관리형 서비스를 사용하면 컨트롤 플레인 관리 부담을 줄일 수 있습니다.

| 서비스 | 클라우드 | 특징 |
|--------|---------|------|
| **EKS** | AWS | AWS 서비스와 통합, Fargate 지원 |
| **GKE** | GCP | Autopilot 모드로 노드 관리 자동화 |
| **AKS** | Azure | Azure AD 통합, 무료 컨트롤 플레인 |

### 6.2 ECS와의 선택 기준

```
AWS 중심 + 단순한 운영       → ECS
멀티 클라우드 + K8s 경험 있음 → EKS
커스텀 네트워킹/서비스 메시   → EKS
```

### 6.3 핵심 도구

| 도구 | 용도 |
|------|------|
| **kubectl** | 클러스터 관리 CLI |
| **Helm** | 쿠버네티스 패키지 매니저 |
| **Kustomize** | YAML 매니페스트 커스터마이징 |
| **Lens / k9s** | 클러스터 모니터링 GUI/TUI |
