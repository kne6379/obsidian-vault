# ngrok

> 로컬 개발 서버를 인터넷에 안전하게 노출하는 보안 터널링 도구입니다.

---

## 1. 정의

ngrok은 로컬 머신에서 실행 중인 서비스를 공개 URL을 통해 인터넷에 노출하는 리버스 프록시 터널링 도구입니다. 별도의 네트워크 설정이나 방화벽 변경 없이, 로컬 서버에 외부에서 접근 가능한 엔드포인트를 생성합니다.

최근에는 단순 터널링을 넘어 API 게이트웨이, 트래픽 관측, DDoS 방어 등의 기능을 제공하는 통합 인그레스 플랫폼으로 발전하고 있습니다.

---

## 2. 등장 배경 및 필요성

로컬 개발 환경에서 외부 서비스와 통신해야 하는 상황이 빈번하지만, 기존 방식에는 여러 문제가 있었습니다.

- **웹훅 수신 불가**: Stripe, GitHub, Slack 등 외부 서비스의 웹훅을 로컬에서 직접 수신할 수 없습니다. 콜백 URL로 `localhost`를 지정할 수 없기 때문에, 별도의 스테이징 서버 배포가 필요했습니다.
- **네트워크 설정의 복잡성**: 포트 포워딩, 방화벽 규칙 변경, 동적 DNS 설정 등 로컬 서버를 외부에 노출하려면 네트워크 지식과 번거로운 설정이 필요합니다.
- **프로토타입 공유의 비효율**: 개발 중인 기능을 팀원이나 고객에게 시연하려면 매번 배포해야 했습니다. 빠른 피드백 루프가 어려웠습니다.

## 3. 작동 원리 / 핵심 개념

### 3.1 아키텍처

ngrok은 에이전트(Agent)와 클라우드 서비스(Cloud Service) 두 가지 핵심 구성 요소로 동작합니다.

```
[인터넷 클라이언트]
        │
        ▼
[ngrok 클라우드 서비스]  ← 공개 엔드포인트 (https://xxxx.ngrok-free.app)
        │
        │  TLS 암호화 터널 (아웃바운드 연결)
        │
        ▼
[ngrok 에이전트]  ←→  [로컬 서비스 (localhost:3000)]
```

1. 로컬 머신에서 ngrok 에이전트를 실행하면, 에이전트가 ngrok 클라우드 서비스에 **아웃바운드 TLS 연결**을 수립합니다.
2. ngrok 클라우드 서비스는 공개 URL(엔드포인트)을 생성하여 외부 트래픽을 수신합니다.
3. 수신된 트래픽은 기존 TLS 연결을 통해 에이전트로 전달되고, 에이전트가 로컬 서비스로 전달합니다.

아웃바운드 연결 방식이므로 방화벽이나 NAT 설정을 변경할 필요가 없습니다.

### 3.2 에이전트 배포 형태

ngrok 에이전트는 다양한 형태로 배포할 수 있습니다.

| 형태 | 용도 |
|------|------|
| CLI | 개발 단계에서 대화형 사용 |
| OS 서비스 | 백그라운드 데몬으로 상시 실행 |
| SDK 라이브러리 | 애플리케이션에 내장하여 프로그래밍 방식으로 제어 |
| 쿠버네티스 오퍼레이터 | [[쿠버네티스]] 클러스터 내 인그레스로 활용 |

### 3.3 모듈 시스템

ngrok 클라우드 서비스와 에이전트 사이에는 모듈(Module)이 위치하여 트래픽을 처리합니다.

- **인증**: OAuth, OIDC, SAML 등 인증 미들웨어
- **트래픽 변환**: 헤더 조작, 요청/응답 변환
- **관측**: 요청 로그, 메트릭 수집
- **보안**: IP 제한, DDoS 방화벽

이 구조는 [[API Gateway]]의 역할과 유사하며, ngrok은 이를 터널링과 결합하여 제공합니다.

### 3.4 기본 사용법

```bash
# 설치 (macOS)
brew install ngrok

# 인증 토큰 설정
ngrok config add-authtoken <토큰>

# 로컬 3000번 포트를 인터넷에 노출
ngrok http 3000
```

실행 후 `https://xxxx.ngrok-free.app` 형태의 공개 URL이 생성됩니다.

## 4. 장점 및 이점

- **즉시 사용 가능**: 한 줄의 명령어로 로컬 서버를 인터넷에 노출할 수 있습니다. DNS, 인증서, 포트 포워딩을 자동으로 처리합니다.
- **네트워크 설정 불필요**: 아웃바운드 TLS 연결 방식이므로 방화벽, NAT, 라우터 설정을 변경할 필요가 없습니다.
- **HTTPS 자동 적용**: 생성되는 공개 URL에 TLS 인증서가 자동으로 적용되어, 별도의 SSL 설정 없이 보안 통신이 가능합니다.
- **트래픽 검사**: 웹 인터페이스(`http://localhost:4040`)를 통해 터널을 통과하는 모든 요청/응답을 실시간으로 확인하고 디버깅할 수 있습니다.
- **다양한 프로토콜 지원**: HTTP/HTTPS뿐 아니라 TCP, TLS 터널도 지원하여 데이터베이스, SSH 등 다양한 서비스를 노출할 수 있습니다.

---

## 5. 한계점 및 고려사항

- **무료 플랜 제약**: 무료 플랜에서는 세션이 2시간으로 제한되며, 월 1GB 대역폭 제한이 있습니다. 동시에 사용 가능한 엔드포인트도 최대 3개로 제한됩니다.
- **URL 변동**: 무료 플랜에서는 터널 재시작 시 URL이 변경됩니다. 고정 도메인이 필요하면 유료 플랜을 사용해야 합니다.
- **보안 위험**: 로컬 서비스를 인터넷에 노출하므로, 인증이 없는 서비스를 열어두면 보안 사고로 이어질 수 있습니다. 반드시 접근 제어를 설정해야 합니다.
- **외부 의존성**: ngrok 클라우드 서비스에 의존하므로, 서비스 장애 시 터널이 중단됩니다. 프로덕션 환경에서 단독 인그레스로 사용하기에는 위험이 있습니다.
- **리전 제한**: 무료 플랜에서는 선택 가능한 서버 리전이 제한되어, 지리적으로 먼 리전에 연결될 경우 지연 시간이 증가할 수 있습니다.

## 6. 실무 적용 가이드

### 6.1 적용 시나리오

| 시나리오 | 설명 |
|----------|------|
| 웹훅 개발 | Stripe, GitHub 등의 웹훅을 로컬에서 수신하여 디버깅 |
| 프로토타입 시연 | 배포 없이 로컬 서비스를 고객이나 팀원에게 공유 |
| 모바일 앱 테스트 | 모바일 기기에서 로컬 백엔드 API에 접근 |
| IoT 디바이스 연동 | 외부 디바이스에서 로컬 서버로 데이터 전송 |

### 6.2 대안 비교

| 도구 | 특징 | 비용 |
|------|------|------|
| ngrok | 풍부한 기능, 트래픽 검사, API 게이트웨이 | 무료 (제한적) / 유료 $8~/월 |
| Cloudflare Tunnel | 대역폭 무제한, Cloudflare 생태계 통합 | 무료 |
| localtunnel | npm 패키지, 간단한 사용 | 무료 (유지보수 중단 상태) |
| Tailscale Funnel | Tailscale 네트워크 기반, P2P 연결 | 무료 (Tailscale 필요) |
