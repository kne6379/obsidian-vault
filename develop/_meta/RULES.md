# 개발 문서 작성 규칙

이 문서는 AI(Claude)가 develop 폴더 내 문서를 작성하거나 수정할 때 따라야 할 규칙을 정의합니다. 문서 작성 전 반드시 이 규칙을 참조하여 일관된 품질을 유지합니다.

---

## 1. 문서 유형별 구조

### 1.1 개념 문서 (concepts/)

개념/기술을 설명하는 문서는 다음 섹션을 순서대로 포함합니다.

1. **정의** - 한 문장 요약 후 상세 설명
2. **등장 배경 및 필요성** - 왜 이 기술/개념이 필요한가
3. **작동 원리 / 핵심 개념** - 어떻게 동작하는가
4. **장점 및 이점** - 도입 시 얻는 효과
5. **한계점 및 고려사항** - 주의할 점, 단점
6. **실무 적용 가이드** (선택) - 구체적 적용 방법
7. **관련 문서** - Obsidian 내부 링크

### 1.2 튜토리얼 문서 (tutorials/)

실습/가이드 문서는 다음 구조를 따릅니다.

1. **개요** - 무엇을 배우는가, 예상 소요 시간
2. **사전 준비** - 필요한 환경, 선행 지식
3. **단계별 진행** - 번호가 매겨진 순차적 단계
4. **결과 확인** - 예상 결과물, 검증 방법
5. **트러블슈팅** - 자주 발생하는 문제와 해결책
6. **관련 문서**

### 1.3 문제 해결 기록 (troubleshooting/)

1. **문제 상황** - 에러 메시지, 증상
2. **원인 분석** - 왜 발생했는가
3. **해결 방법** - 구체적 해결 단계
4. **예방 방법** (선택) - 재발 방지책

## 2. 작성 스타일

### 2.1 문체

- 모든 문서는 **"~니다" 체의 공식 문어체**로 작성합니다.
- 구어체, 반말, 이모지는 사용하지 않습니다.

### 2.2 용어 사용

- **한글 우선 원칙**: 의미 전달에 불필요한 영어 단어의 괄호 병기는 지양합니다.
- **영문 병기 기준**: 최초 등장 시 개념 정의가 필요한 경우에만 `한글(English)` 형식으로 병기합니다.
- 이후 동일 문서 내에서는 한글만 사용합니다.
- 용어 정의는 [[GLOSSARY]]를 참조하고, 새 용어는 용어집에 추가합니다.

### 2.3 분량 및 구조

- 핵심 정보가 누락되지 않도록 하되, 불필요하게 장황한 설명은 피합니다.
- 긴 설명이 필요한 경우 문단 구분 또는 글머리 기호를 활용합니다.
- 한 섹션이 지나치게 길어지면 하위 섹션으로 분리합니다.

### 2.4 명확성

- 추측이나 모호한 표현은 피합니다.
- 정의, 구분, 비교가 필요한 경우 명확히 구분하여 설명합니다.
- "~일 수 있습니다", "~인 것 같습니다" 등의 표현은 최소화합니다.

## 3. 메타데이터 (YAML Frontmatter)

모든 문서 최상단에 다음 형식의 프론트매터를 포함합니다.

```yaml
---
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [태그1, 태그2]
status: draft | review | done
---
```

### 3.1 태그 분류 체계

**유형 태그**
- `concept` - 개념/이론 문서
- `tutorial` - 실습/가이드 문서
- `troubleshooting` - 문제 해결 기록
- `reference` - 참고 자료

**주제 태그**
- `ai`, `llm`, `rag` - AI/머신러닝 관련
- `devops`, `docker`, `k8s` - 인프라/배포 관련
- `backend`, `frontend` - 개발 영역
- `database`, `api` - 기술 스택

**상태 태그**
- `draft` - 초안 작성 중
- `review` - 검토 필요
- `done` - 완료

## 4. 링크 규칙

### 4.1 내부 링크

- 다른 문서에서 정의된 개념은 **`[[문서명]]`** 형식으로 링크합니다.
- 특정 섹션 링크: `[[문서명#섹션명]]`
- 별칭 사용: `[[문서명|표시할 텍스트]]`

### 4.2 관련 문서 섹션

모든 문서 하단에 `## 관련 문서` 섹션을 추가하고, 관련 문서들을 링크합니다.

```markdown
## 관련 문서

- [[관련개념1]] - 간단한 관계 설명
- [[관련개념2]] - 간단한 관계 설명
```

### 4.3 외부 링크

외부 참고 자료는 `## 참고 자료` 섹션에 별도 정리합니다.

```markdown
## 참고 자료

- [제목](URL) - 간단한 설명
```

## 5. 품질 기준

### 5.1 필수 충족 조건

- [ ] 프론트매터가 올바른 형식으로 포함되어 있는가
- [ ] 문서 유형에 맞는 섹션 구조를 따르는가
- [ ] 비전문가도 이해할 수 있는 수준의 설명이 포함되어 있는가
- [ ] 실무 관점의 장단점/고려사항이 포함되어 있는가
- [ ] 관련 문서 링크가 적절히 연결되어 있는가

### 5.2 지양 사항

- 추측성 표현 ("~인 것 같습니다", "아마도")
- 과도한 영문 병기
- 근거 없는 주장
- 지나치게 짧거나 긴 설명

---

## 6. 문서 작성 워크플로우

1. **템플릿 복사**: `_templates/` 폴더에서 적절한 템플릿을 복사합니다.
2. **프론트매터 작성**: 날짜, 태그, 상태를 설정합니다.
3. **내용 작성**: 섹션 구조에 따라 내용을 채웁니다.
4. **링크 연결**: 관련 문서를 [[링크]]로 연결합니다.
5. **용어 확인**: 새 용어가 있으면 [[GLOSSARY]]에 추가합니다.
6. **INDEX 업데이트**: [[INDEX]]에 새 문서를 등록합니다.
7. **상태 변경**: 완료 시 `status: done`으로 변경합니다.
