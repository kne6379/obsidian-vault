# projects 영역 규칙

## 문서 유형별 섹션 구조

### 프로젝트 문서 (projects/)

프로젝트의 전체 맥락을 기록하는 문서입니다.

1. 개요 — 프로젝트 목적, 배경, 기간
2. 기술 스택 — 사용 기술과 선택 사유
3. 아키텍처 — 시스템 구조, 주요 컴포넌트
4. 주요 기능 — 핵심 기능 설명
5. 이슈 및 회고 (선택) — 프로젝트 중 발생한 주요 이슈, 배운 점
6. 관련 문서

### 기술 의사결정 기록 (decisions/)

기술 선택의 배경과 근거를 보존하는 문서입니다.

1. 배경 — 의사결정이 필요했던 상황
2. 검토 대안 — 비교한 선택지와 각각의 장단점
3. 최종 결정 — 선택한 대안과 그 사유
4. 관련 문서

### 노트 (notes/)

미팅 노트, 리서치 정리 등 공유 목적의 문서입니다. 태그로 미팅/리서치를 구분합니다.

1. 요약 — 핵심 내용 요약
2. 상세 내용 — 논의 사항, 조사 결과 등
3. 액션 아이템 (선택) — 후속 작업이 있는 경우
4. 관련 문서

### 문제 해결 기록 (troubleshooting/)

1. 문제 상황 — 에러 메시지, 증상
2. 원인 분석 — 왜 발생했는가
3. 해결 방법 — 구체적 해결 단계
4. 예방 방법 (선택) — 재발 방지책

## 상태값

| 상태 | 설명 |
|------|------|
| `draft` | 초안 작성 중 |
| `active` | 진행 중인 프로젝트 |
| `completed` | 완료된 프로젝트 |
| `archived` | 더 이상 관리하지 않는 문서 |

## 태그 체계

- **유형**: project, decision, note, troubleshooting, reference
- **노트 세부**: meeting, research
- **상태**: draft, active, completed, archived

## 작업 전 필수 확인

- 새 프로젝트 문서 작성 시 `projects/_templates/PROJECT.md` 템플릿을 기반으로 합니다.
- 새 의사결정 문서 작성 시 `projects/_templates/DECISION.md` 템플릿을 기반으로 합니다.
- 새 노트 작성 시 `projects/_templates/NOTE.md` 템플릿을 기반으로 합니다.
- 새 문서 작성 전 반드시 `projects/_meta/INDEX.md`를 읽어 중복을 방지합니다.
- 새 용어 작성 시 `projects/_meta/GLOSSARY.md`를 확인합니다.
