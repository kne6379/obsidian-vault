# Multi-Agent Analyze Skill Design

> **For agentic workers:** This is a design spec, not an implementation plan.

**Goal:** 기존 `/analyze` 스킬을 5개 전문가 에이전트 기반의 삶 전반 분석 시스템으로 업그레이드

**Summary:** 볼트 구조 분석가가 먼저 팩트 데이터를 수집하고, 4개 전문가 에이전트(커리어/기술, 사업 전략, 삶의 멘토, 트렌드 분석)가 병렬로 각자 영역을 분석한 뒤, 메인 세션이 형 페르소나로 종합 조언을 작성하여 `_meta/ANALYSIS.md`에 저장한다.

---

## 1. 결정 사항 요약

| 항목 | 결정 |
|------|------|
| 기존 analyze와의 관계 | 대체 (기존 analyze를 업그레이드) |
| 에이전트 수 | 5개 (구조 분석 + 커리어 + 사업 + 삶 + 트렌드) |
| 실행 순서 | 순차-병렬 파이프라인 (1단계 단독 → 2단계 4개 병렬 → 3단계 취합) |
| 취합 방식 | 메인 세션이 형 페르소나로 직접 |
| 출력물 | 단일 파일 `_meta/ANALYSIS.md` |
| 웹 서치 범위 | 기술 + 시장 + 커리어 전방위 |
| 경량 모드 | 없음. 항상 풀 분석 |
| 에이전트 모델 | 전부 opus |

---

## 2. 실행 아키텍처

```
/analyze 실행
    │
    ▼
[1단계] 볼트 구조 분석가 (Agent - Explore, opus)
    │  단독 실행
    │  → 구조 분석 리포트 출력
    │
    ▼
[2단계] 4개 전문가 에이전트 (Agent, opus, 병렬)
    │
    ├─ 커리어/기술 전문가 (general-purpose)
    ├─ 사업 전략가 (general-purpose)
    ├─ 삶의 멘토 (general-purpose)
    └─ 트렌드 분석가 (general-purpose)
    │
    ▼
[3단계] 메인 세션 취합
    • 5개 결과 종합
    • 형 페르소나로 최종 조언
    • _meta/ANALYSIS.md 저장
```

---

## 3. 에이전트 상세 설계

### 3.1 볼트 구조 분석가

**타입**: Explore (읽기 전용 탐색)
**모델**: opus

**입력**: 없음 (볼트 직접 스캔)

**수행 작업**:
1. 4개 영역의 인덱스 파일 읽기: `knowledge/_meta/INDEX.md`, `work/_meta/INDEX.md`, `life/_meta/INDEX.md`, `ventures/_meta/INDEX.md`
2. 등록된 문서 목록 추출
3. 실제 파일 존재 확인 (Glob)
4. 미등록 문서 탐지 (INDEX에 없지만 파일 존재)
5. 각 문서 프론트매터 파싱: status, tags, created, updated
6. 상태별 분류, 오래된 draft 탐지 (7일 이상)
7. `[[링크]]` 패턴 추출 → 그래프 구축
8. 허브 문서, 고립 문서, 끊어진 링크, 영역 간 연결 분석
9. 태그 분포 집계
10. 각 영역 문서 제목/상태/태그 요약 목록 작성 (2단계 컨텍스트용)

**출력**: 구조화된 텍스트 리포트
- 영역별 통계
- 문서 목록 (제목, 상태, 태그, 날짜)
- 그래프 분석 결과
- 태그 분포
- 관심사 추정

---

### 3.2 커리어/기술 전문가

**타입**: general-purpose
**모델**: opus

**페르소나**: 10년차 시니어 엔지니어 겸 테크 리드. 기술 선택과 커리어 전략에 밝다.

**입력**:
- 1단계 구조 리포트 (문서 목록, 태그, 상태 포함 — 에이전트가 직접 탐색하지 않고 리포트에서 추출)

**수행 작업**:
1. knowledge/ 문서 읽기: 어떤 기술을 정리했는지 파악
2. work/ 프로젝트 문서 읽기: 실무에서 쓰는 기술 파악
3. 학습 간극 분석: 실무에서 쓰지만 정리 안 한 기술, 정리했지만 실무에 안 쓰는 기술
4. 웹 서치: 백엔드/AI 엔지니어 채용 트렌드, 요구 기술, 연봉 동향
5. 현재 포지셔닝 평가: 시장이 원하는 것 vs 갖고 있는 것
6. 학습 우선순위 제안

**출력**:
- 현재 기술 역량 맵
- 학습 간극 (알아야 하는데 모르는 것)
- 채용 시장 동향과 포지셔닝
- 학습 우선순위 제안 (구체적으로)

---

### 3.3 사업 전략가

**타입**: general-purpose
**모델**: opus

**페르소나**: 스타트업 경험 있는 사업 개발 전문가. 실현 가능성 중시.

**입력**:
- 1단계 구조 리포트 (ventures 영역 문서 목록, 상태, 태그 포함)

**수행 작업**:
1. ventures/ 브레인스토밍 문서 읽기: 아이디어 내용, 평가 점수, 상태
2. 웹 서치: 해당 도메인(AI 콘텐츠 생성, n8n 등) 최신 시장 동향
3. 경쟁 환경 변화 체크: 새로운 플레이어, 기존 플레이어 변화
4. 아이디어별 현재 유효성 재평가
5. 다음 액션 제안: 어떤 아이디어를 밀고, 어떤 건 접을지

**출력**:
- 아이디어 진행도 평가
- 시장 기회/위험 업데이트
- 다음 액션 제안 (구체적인 검증 단계 포함)

---

### 3.4 삶의 멘토

**타입**: general-purpose
**모델**: opus

**페르소나**: 삶의 경험이 풍부한 현자. 진솔하고 따뜻하되 핵심을 찌른다.

**입력**:
- 1단계 구조 리포트 (life 영역 문서 목록, 전체 활동 패턴 포함)

**수행 작업**:
1. life/ 문서 전체 읽기: 가치관, 의사결정, 자기관리 문서
2. 활동 패턴 분석: 볼트 전체에서 뭘 많이 쓰고 뭘 안 쓰는지 (구조 리포트 기반)
3. 가치관 정합성 체크: 말하는 것(life 문서)과 하는 것(knowledge/work 활동)이 일치하는지
4. 삶의 균형 평가: 일/학습/사업/개인성장 비율
5. 내면 성장 방향 제안

**웹 서치**: 하지 않음 (개인 내면 영역)

**출력**:
- 가치관 정합성 체크
- 삶의 균형 평가
- 내면 성장 방향 조언

**주의**: life/ 영역은 민감한 개인 기록이므로, 판단이 아닌 관찰과 질문 형태로 조언한다.

---

### 3.5 트렌드 분석가

**타입**: general-purpose
**모델**: opus

**페르소나**: 테크/비즈니스 리서처. 넓게 보고 기회를 포착한다.

**입력**:
- 1단계 구조 리포트 (태그 분포, 관심사 추정)

**수행 작업**:
1. 구조 리포트에서 관심사 키워드 추출
2. 웹 서치 (3-5회):
   - AI/LLM 최신 기술 동향 (모델, 프레임워크, 도구)
   - AI SaaS/자동화 시장 동향
   - 백엔드/DevOps 기술 트렌드
   - 개발자 커리어 시장 동향
3. 볼트에 없는 빈 영역 탐지: 트렌드 상 중요한데 아직 정리 안 한 주제
4. 기회 포착: 관심사와 트렌드의 교차점

**출력**:
- 주목할 트렌드 3-5개 (각각 왜 중요한지, 볼트와 어떻게 연결되는지)
- 볼트 확장 제안 (정리하면 좋을 주제)
- 기회 포착 포인트

---

## 4. 출력물 구조

볼트 루트의 `_meta/ANALYSIS.md` (절대 경로: `{vault_root}/_meta/ANALYSIS.md`):

```markdown
---
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [meta, analysis]
status: done
---

# 볼트 분석 결과

**분석 일시**: YYYY-MM-DD

---

## 1. 볼트 현황
(볼트 구조 분석가 결과 — 팩트 데이터)

## 2. 커리어/기술 분석
(커리어/기술 전문가 결과)

## 3. 사업 분석
(사업 전략가 결과)

## 4. 삶의 방향
(삶의 멘토 결과)

## 5. 트렌드 브리핑
(트렌드 분석가 결과)

## 6. 종합 조언
(메인 세션 — 형 페르소나, 반말)
- 당장 해야 할 것
- 이번 주/이번 달 방향
- 전체 평가 (잘한 것, 개선할 것)
- 한마디

---

## 분석 히스토리
| 날짜 | 비고 |
|------|------|
```

- 섹션 1은 팩트(데이터), 섹션 2-5는 각 전문가의 해석과 조언
- 섹션 6은 메인 세션이 전부 종합해서 형 페르소나로 최종 조언
- 히스토리는 날짜+비고만 간결하게 (이전 분석 전문은 git 히스토리로 보존)
- 저장 시 기존 파일의 분석 히스토리 테이블을 읽어서 새 행을 추가한 뒤 전체 파일을 덮어쓴다

## 5. 스킬 파일 구조

기존 `.claude/commands/analyze.md` 파일을 교체한다.

**기존 기능 변경**: `/analyze develop`, `/analyze --brief` 등 인자 기반 옵션은 모두 제거한다. `$ARGUMENTS`는 무시한다. `/analyze`는 항상 전체 풀 분석을 실행한다.

**스킬 파일 작동 방식**: 스킬 파일은 자연어 지시문으로 구성된다. Claude Code가 스킬을 로드하면 해당 지시를 따라 Agent 도구를 호출한다. 각 에이전트 프롬프트는 스킬 파일 내에 인라인으로 포함된다.

스킬 파일에 포함될 내용:
1. 프론트매터 (`description`)
2. 실행 순서 (1단계 → 2단계 → 3단계)
3. 각 에이전트에 전달할 프롬프트 (인라인 자연어 지시)
4. 출력물 구조 명세
5. 종합 조언 작성 가이드 (형 페르소나 규칙)

**2단계 에이전트 제약**: general-purpose 에이전트는 쓰기 권한을 가지지만, 프롬프트에서 "읽기만 수행하고 파일을 수정하지 마라"고 명시적으로 제약한다.

---

## 6. 제약 사항

- 에이전트는 볼트 문서를 **읽기만** 한다. 수정하지 않는다.
- 유일한 쓰기 작업은 메인 세션의 `_meta/ANALYSIS.md` 저장뿐이다.
- life/ 영역 문서는 에이전트가 읽되, 출력에 민감한 내용을 직접 인용하지 않고 요약만 한다.
- 웹 서치는 커리어/기술, 사업 전략, 트렌드 에이전트만 수행한다. 삶의 멘토는 하지 않는다.
- 분석 히스토리에서 이전 분석 전문은 저장하지 않는다. git diff로 비교 가능하다.
- 에이전트 하나가 실패하거나 웹 서치가 안 되는 경우, 해당 섹션에 "분석 불가 (사유)" 를 표시하고 나머지 섹션은 정상 진행한다.
- 웹 서치는 에이전트당 3-5회로 제한한다.

---

## 7. 연관 파일 업데이트

- 기존 CLAUDE.md의 "분석 커맨드" 섹션은 이미 제거되어 있으므로 추가 변경 불필요하다.
- `.claude/commands/analyze.md` 파일만 교체하면 된다.
