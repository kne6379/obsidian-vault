# Claude 스킬 시스템

> Claude가 특정 워크플로우를 일관되게 수행하도록 지침을 폴더 단위로 패키징하는 기능으로, [[CLAUDE.md 설계 원칙|CLAUDE.md]]가 "무엇을 지킬지"를 정의한다면 스킬은 "어떻게 수행할지"를 정의합니다.

---

## 1. 정의

스킬(Agent Skill)은 Claude에게 특정 작업이나 워크플로우를 가르치는 **지침 패키지**입니다. 하나의 폴더에 마크다운 지침, 실행 스크립트, 참고 문서, 템플릿을 묶어 배포하며, Claude.ai, Claude Code, API 전 환경에서 동일하게 동작합니다.

[[CLAUDE.md 설계 원칙|CLAUDE.md]]가 프로젝트 전반의 규칙을 **항상** 주입하는 메커니즘이라면, 스킬은 특정 상황에서만 **온디맨드로** 활성화되는 전문 지침입니다. 두 메커니즘은 상호 보완적이며, 함께 사용할 때 세션 간 일관성과 작업 전문성을 모두 확보할 수 있습니다.

---

## 2. 등장 배경 및 필요성

- **반복 지시의 비효율**: 동일한 워크플로우를 매 세션마다 프롬프트로 설명하면 토큰이 낭비되고 결과 편차가 발생합니다. 스킬은 한 번 작성하면 모든 세션에서 재사용됩니다.
- **도구 접근과 활용 지식의 분리**: MCP가 외부 서비스에 대한 접근(도구)을 제공하더라도, 그 도구를 어떤 순서와 맥락으로 사용해야 하는지(지식)는 별도로 전달해야 합니다. 스킬은 이 지식 계층을 담당합니다.
- **조직 표준화 요구**: 팀 전체가 Claude를 동일한 방식으로 활용하려면, 개인 프롬프트가 아닌 공유 가능한 형태의 워크플로우 정의가 필요합니다.

---

## 3. 작동 원리 / 핵심 개념

### 3.1 파일 구조

스킬은 하나의 폴더로 구성되며, 폴더명은 반드시 케밥 케이스(kebab-case)를 사용합니다.

```
your-skill-name/
├── SKILL.md          # 필수 — 메인 지침 파일 (대소문자 정확히)
├── scripts/          # 선택 — 실행 코드 (Python, Bash 등)
├── references/       # 선택 — API 가이드, 예시 등 참고 문서
└── assets/           # 선택 — 템플릿, 폰트, 아이콘
```

`SKILL.md`는 YAML 프론트매터와 마크다운 본문으로 구성됩니다.

```yaml
---
name: your-skill-name           # 필수, kebab-case
description: 무엇을 하는지 + 언제 사용하는지  # 필수, 1024자 이내
license: MIT                    # 선택
metadata:                       # 선택
  author: Company
  version: 1.0.0
  mcp-server: server-name
---
```

**프론트매터 규칙:**

| 항목 | 규칙 |
|------|------|
| `name` | kebab-case만 허용, 폴더명과 일치, "claude"/"anthropic" 사용 금지 |
| `description` | 기능 설명 + 트리거 조건을 반드시 포함 |
| 금지 사항 | XML 태그(`< >`), 폴더 내 README.md |

### 3.2 점진적 공개

스킬은 3단계 로딩 체계를 사용하여 [[GLOSSARY#컨텍스트 윈도우|컨텍스트 윈도우]]를 효율적으로 관리합니다.

| 단계 | 대상 | 로딩 시점 |
|------|------|-----------|
| 1단계 | YAML 프론트매터 | 항상 (시스템 프롬프트에 포함) |
| 2단계 | SKILL.md 본문 | 스킬이 관련 있다고 판단될 때 |
| 3단계 | references/, scripts/ 등 | Claude가 필요하다고 판단할 때 |

이 구조는 [[CLAUDE.md 설계 원칙#3.1 로딩 메커니즘|CLAUDE.md의 로딩 메커니즘]]과 유사하지만, 스킬은 1단계에서 프론트매터만 노출하여 토큰 소비를 최소화합니다.

### 3.3 사용 사례 카테고리

Anthropic은 스킬의 용도를 세 가지 카테고리로 구분합니다.

| 카테고리 | 용도 | 핵심 기법 |
|----------|------|-----------|
| 문서/자산 생성 | 일관된 결과물 생산 (문서, 코드, 디자인) | 스타일 가이드 내장, 템플릿, 품질 체크리스트 |
| 워크플로우 자동화 | 다단계 프로세스의 체계적 실행 | 단계별 검증, 반복 개선 루프 |
| MCP 강화 | MCP 도구 활용에 전문 지식 추가 | 다중 MCP 호출 조율, 에러 처리 |

### 3.4 설계 패턴

가이드에서 제시하는 5가지 검증된 패턴입니다.

**패턴 1: 순차 워크플로우** — 단계별 실행 순서가 중요한 프로세스에 적합합니다. 각 단계 간 의존성, 검증, 롤백 지침을 명시합니다.

**패턴 2: 다중 MCP 조율** — 여러 서비스를 넘나드는 워크플로우입니다. 단계(Phase)별 분리, 서비스 간 데이터 전달, 단계별 검증이 핵심입니다.

**패턴 3: 반복 개선** — 초안 생성 → 검증 스크립트 실행 → 문제 식별 → 수정 → 재검증의 루프를 반복하여 품질을 높입니다.

**패턴 4: 맥락 기반 도구 선택** — 동일한 목표라도 파일 크기, 유형, 상황에 따라 다른 도구를 선택하는 의사결정 트리를 내장합니다.

**패턴 5: 도메인 전문 지식** — 컴플라이언스 검사, 감사 추적 등 도구 접근 외의 전문 지식을 스킬에 임베딩합니다.

### 3.5 MCP와의 관계

MCP(Model Context Protocol)와 스킬은 상호 보완적입니다.

| 구분 | MCP | 스킬 |
|------|-----|------|
| 역할 | 도구 접근 (연결) | 활용 지침 (지식) |
| 비유 | 전문 주방 (도구, 재료, 장비) | 레시피 (조리법) |
| 제공하는 것 | Claude가 **무엇을** 할 수 있는지 | Claude가 **어떻게** 해야 하는지 |

MCP만 있으면 사용자가 매번 활용법을 설명해야 하고, 스킬을 함께 제공하면 워크플로우가 자동으로 활성화됩니다.

---

## 4. 장점 및 이점

- **세션 간 일관성**: 동일한 워크플로우가 매번 같은 품질과 절차로 실행됩니다. 프롬프트 편차에 의한 결과 변동이 사라집니다.
- **토큰 효율성**: 점진적 공개로 관련 없는 스킬은 프론트매터만 로드되어 [[GLOSSARY#컨텍스트 윈도우|컨텍스트 윈도우]]를 절약합니다.
- **이식성**: Claude.ai, Claude Code, API 전 환경에서 수정 없이 동작합니다. 오픈 표준으로 발표되어 다른 AI 플랫폼에서도 활용 가능합니다.
- **조직 배포**: 관리자가 워크스페이스 전체에 스킬을 배포하여 팀 단위 표준화가 가능합니다.
- **MCP 시너지**: 원시 도구 접근에 전문 지식을 결합하여 사용자의 학습 곡선을 낮춥니다.

---

## 5. 한계점 및 고려사항

- **트리거 정밀도**: description 작성이 부정확하면 관련 없는 쿼리에 스킬이 활성화(오버트리거링)되거나, 필요한 상황에서 활성화되지 않는(언더트리거링) 문제가 발생합니다. 부정 트리거("이런 경우에는 사용하지 않는다")를 적극 활용해야 합니다.
- **SKILL.md 크기 제한**: 본문이 5,000단어를 초과하면 성능이 저하됩니다. 상세 문서는 반드시 `references/`로 분리하여 점진적 공개를 활용해야 합니다.
- **동시 활성 스킬 수**: 20~50개 이상의 스킬이 동시에 활성화되면 프론트매터만으로도 상당한 토큰을 소비합니다.
- **지침의 비결정성**: 언어 기반 지침은 코드처럼 결정적으로 동작하지 않습니다. 중요한 검증 로직은 `scripts/` 폴더에 코드로 구현하는 것이 권장됩니다.
- **테스트 도구 미성숙**: 정량적 평가 도구가 아직 발전 중이며, 현재는 수동 테스트와 정성적 평가에 의존하는 부분이 있습니다.

## 6. 실무 적용 가이드

### 6.1 현재 볼트의 설정 구조

이 볼트의 Claude Code 설정은 스킬 가이드의 개념을 이미 상당 부분 적용하고 있습니다.

**글로벌 플러그인 (`~/.claude/settings.json`)**

| 플러그인 | 역할 | 가이드 대응 |
|----------|------|-------------|
| superpowers | 14개 개발 워크플로우 스킬 라이브러리 | 카테고리 2 (워크플로우 자동화) |
| code-review | PR 코드 리뷰 | 카테고리 2 |
| skill-creator | 스킬 생성/리뷰 도구 | 가이드에서도 권장하는 도구 |
| discord | 디스코드 연동 | 카테고리 3 (MCP 강화) |

**프로젝트 커맨드 (`.claude/commands/`)**

| 커맨드 | 역할 | 가이드 패턴 대응 |
|--------|------|------------------|
| `/analyze` | 멀티에이전트 볼트 분석 | 패턴 2 (다중 조율) + 패턴 5 (도메인 전문 지식) |
| `/write` | 개념 문서 작성 + 검수 + 등록 | 패턴 1 (순차 워크플로우) + 패턴 3 (반복 개선) |
| `/brainstorm` | AI 사업 브레인스토밍 | 패턴 1 (순차 워크플로우) |

**프로젝트 규칙 (`.claude/rules/`)**

| 규칙 | 적용 범위 |
|------|-----------|
| `common.md` | 전체 — 문체, 용어, 프론트매터, 링크 규칙 |
| `knowledge.md` | knowledge/ — 문서 유형별 구조, 태그 체계 |
| `ventures.md` | ventures/ — 사업 기획 구조, 평가 프레임워크 |
| `projects.md` | projects/ — 프로젝트/의사결정 문서 구조 |
| `life.md` | life/ — 회고, 의사결정, 가치관 기록 |

### 6.2 가이드 기준 점검 결과

Anthropic 가이드의 주요 원칙을 기준으로 현재 설정을 점검한 결과입니다.

**잘 적용된 부분:**

- **점진적 공개**: `.claude/rules/`에서 영역별 규칙을 분리하여, 해당 영역 파일 작업 시에만 로드되도록 설계되어 있습니다. 가이드의 3단계 로딩 원칙에 부합합니다.
- **구체적 지침**: 각 규칙 파일이 문서 유형별 섹션 구조, 태그 체계, 품질 체크리스트를 명시합니다. 가이드가 경고하는 "모호한 지침" 문제를 회피합니다.
- **설계 패턴 활용**: `/analyze` 커맨드가 다중 에이전트 조율(패턴 2), `/write`가 순차 워크플로우 + 반복 개선(패턴 1+3)을 적용합니다.
- **도메인 지식 내장**: 각 규칙 파일에 해당 영역의 전문 지식(태그 체계, 평가 프레임워크 등)이 임베딩되어 있습니다. 패턴 5에 해당합니다.

**개선 가능한 부분:**

- **description 트리거 조건**: `.claude/commands/` 내 커맨드 파일에는 YAML 프론트매터가 없습니다. 가이드가 강조하는 "언제 사용하는지"(트리거 조건)를 명시하면 Claude의 자동 활성화 정확도가 높아질 수 있습니다.
- **에러 처리 지침 부재**: 현재 커맨드에 실패 시나리오에 대한 안내가 부족합니다. 가이드는 각 단계에서의 에러 처리와 롤백 지침을 권장합니다.
- **검증 스크립트 미활용**: 가이드는 중요한 검증을 언어 지침 대신 `scripts/` 폴더의 코드로 구현할 것을 권장합니다. 현재 볼트의 `scripts/` 디렉토리에 자동화 스크립트가 일부 존재하지만, 커맨드와의 연결이 명시적이지 않습니다.
- **성능 기준선 부재**: 가이드는 "스킬 없이 수행 시 토큰/메시지 수"와 비교하는 성능 기준선 측정을 권장합니다. 현재 커맨드의 효율성을 정량적으로 추적하는 체계가 없습니다.

### 6.3 설정 체계 전체 구조

현재 볼트에서 Claude Code의 동작을 제어하는 설정 파일의 관계를 정리합니다.

```
Claude Code 설정 체계
│
├── 글로벌 설정 (~/.claude/)
│   ├── settings.json          ← 플러그인, 노력 수준, 음성
│   ├── settings.local.json    ← 전역 권한
│   └── plugins/               ← superpowers, code-review 등
│       └── cache/claude-plugins-official/
│           └── superpowers/5.0.5/skills/  ← 14개 개발 스킬
│
├── 프로젝트 설정 (Obsidian Vault/.claude/)
│   ├── settings.local.json    ← 프로젝트별 권한 (git, WebSearch 등)
│   ├── commands/              ← /analyze, /write, /brainstorm
│   └── rules/                 ← 영역별 규칙 5개
│
├── 프로젝트 지시 (Obsidian Vault/CLAUDE.md)
│   └── 페르소나, 볼트 구조 안내
│
└── 세션 메모리 (~/.claude/projects/.../memory/)
    └── MEMORY.md + 개별 메모리 파일
```

이 구조에서 각 계층의 역할은 다음과 같습니다.

| 계층 | 역할 | 가이드 용어 대응 |
|------|------|------------------|
| CLAUDE.md | 프로젝트 전반 규칙 (항상 로드) | - |
| `.claude/rules/` | 영역별 조건부 규칙 | 점진적 공개 2단계 |
| `.claude/commands/` | 명시적 호출 워크플로우 | 스킬 (수동 트리거) |
| superpowers 플러그인 | 자동 활성화 개발 워크플로우 | 스킬 (자동 트리거) |
| 메모리 | 세션 간 학습 지속 | - |
