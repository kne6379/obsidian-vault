# 멀티 AI 에이전트 개발 자동화 워크플로우

> Claude Code, Gemini CLI 두 AI 에이전트를 교차 검증 방식으로 활용하여, 이슈 생성부터 코드리뷰·머지까지 개발 라이프사이클 전체를 자동화하는 1인 개발 워크플로우입니다.

---

## 1. 정의

멀티 AI 에이전트 개발 자동화 워크플로우란, 단일 AI 도구에 의존하지 않고 **복수의 AI 코딩 에이전트를 파이프라인의 각 단계에 배치**하여 개발 프로세스를 자동화하는 체계입니다.

핵심 원칙은 **교차 검증**입니다. 하나의 AI가 생성한 결과물을 다른 AI가 검토함으로써, 단일 모델의 편향이나 환각을 구조적으로 걸러냅니다. 사람은 최종 의사결정과 품질 게이트 역할에 집중합니다.

---

## 2. 등장 배경 및 필요성

[[AI 네이티브 엔지니어]]가 개발 전 과정에 AI를 투입하면서, 다음 문제들이 대두되었습니다.

**단일 AI의 한계**: 하나의 모델이 코드를 작성하고 스스로 리뷰하면, 동일한 맹점을 공유합니다. 모델이 놓친 버그나 아키텍처 결함을 같은 모델이 발견하기 어렵습니다.

**검증 없는 자동화의 위험**: AI가 생성한 코드를 검증 없이 머지하면, 미묘한 논리 오류나 보안 취약점이 프로덕션에 유입됩니다. 사람이 모든 코드를 라인 바이 라인으로 검토하면 자동화의 의미가 퇴색합니다.

**도구별 강점 차이**: Claude Code는 복잡한 추론과 장문 컨텍스트 처리에 강하고, Gemini CLI는 무료 티어와 검색 기반 그라운딩에 특화됩니다. 각 도구의 강점을 적재적소에 배치하면 전체 파이프라인의 품질이 향상됩니다.

## 3. 에이전트별 역할 분담

### 3.1 Claude Code — 주력 개발 에이전트

Anthropic의 CLI 코딩 에이전트입니다. [[CLAUDE.md 설계 원칙]]에 따라 프로젝트별 규칙을 자동 로드하며, 다음 확장 메커니즘을 지원합니다.

| 메커니즘 | 용도 | 위치 |
|----------|------|------|
| CLAUDE.md + `@import` | 프로젝트 규칙 자동 주입, Compaction 생존 | `./CLAUDE.md`, `.claude/rules/` |
| 훅(Hooks) | 이벤트 기반 자동 실행 (커밋 전 린트 등) | `.claude/hooks/` |
| 스킬(Skills) | 재사용 가능한 작업 프롬프트, 슬래시 명령 | `.claude/skills/` |
| MCP 서버 | 외부 도구/데이터 소스 연결 | `.mcp.json` |
| 서브에이전트 | 복잡한 작업의 병렬 분산 처리 | 세션 내 자동 생성 |

**담당 단계**: 이슈 분석, 계획 수립, 코드 작성, 테스트 작성(TDD), 커밋, PR 생성, 리뷰 종합 판단, 피드백 반영

### 3.2 Gemini CLI — 리뷰 에이전트

Google의 오픈소스 터미널 AI 에이전트입니다. 무료 티어에서 분당 60회, 일일 1,000회 요청이 가능하며 100만 토큰 컨텍스트 윈도우를 제공합니다.

| 기능 | 설명 |
|------|------|
| 계획 모드 | 읽기 전용 도구로 의존성 분석 후 실행 계획 수립 |
| 검색 기반 그라운딩 | Google 검색으로 최신 정보 참조 |
| MCP 지원 | GitHub MCP 서버 연결로 PR 직접 코멘트 |
| 무료 대량 호출 | 비용 부담 없이 리뷰 1단계 전담 |

**담당 단계**: 코드리뷰 1단계 (아키텍처/설계 리뷰, 코드 품질/운영 리뷰)

### 3.3 Codex CLI — 선택적 보조 에이전트

OpenAI의 터미널 코딩 에이전트입니다. 서브에이전트 워크플로우로 대규모 작업을 병렬 분산 처리할 수 있습니다. 현재 워크플로우에서는 미도입 상태이며, 향후 테스트 병렬 생성이나 대안 구현 비교 용도로 도입을 검토합니다.

## 4. 전체 파이프라인

```
이슈 생성 → 브랜치 생성 → 계획 수립 → TDD 개발 → 커밋
    → PR 생성 → 코드리뷰(Gemini 1단계 → Claude 2단계)
    → 피드백 반영 → 머지
```

각 단계에서 Claude Code가 주도하고, 코드리뷰 단계에서만 Gemini CLI가 1단계 리뷰를 수행합니다. 사람은 단계 간 전환 시점에서 검토 및 승인합니다.

### 4.1 이슈 → 브랜치

1. GitHub Issues에서 이슈를 생성합니다.
2. Claude Code가 이슈 본문을 분석하고, 영향 범위와 작업량을 파악합니다.
3. 이슈 번호 기반으로 브랜치를 생성합니다 (`feat/15-auth-onboarding` 등).

### 4.2 계획 수립 → TDD 개발

1. Claude Code가 CLAUDE.md와 `.claude/rules/`의 규칙을 자동 로드합니다.
2. 스펙 문서(`docs/domains/`)를 읽고 테스트 케이스를 도출합니다.
3. TDD 사이클을 실행합니다: 테스트 작성 → 실패 확인 → 구현 → 통과 확인.
4. 도메인 규칙 테스트부터 시작하여 유스케이스, 인프라, E2E 순으로 계층별로 진행합니다.

TDD 워크플로우는 `.claude/skills/test-conventions/SKILL.md`에 정의되어 있으며, 다음 계층별 전략을 따릅니다.

| 테스트 레벨 | 위치 | 외부 의존성 |
|------------|------|------------|
| 단위(도메인) | `tests/unit/domain/` | 없음 (순수 Python) |
| 단위(애플리케이션) | `tests/unit/application/` | Port를 Fake로 대체 |
| 통합 | `tests/integration/infrastructure/` | 실제 로컬 DB |
| E2E | `tests/e2e/` | 실제 DB + TestClient |

### 4.3 커밋 → PR 생성

1. Claude Code가 변경 파일을 분석하여 논리적 커밋 단위를 구성합니다.
2. Conventional Commits 형식으로 커밋 메시지를 생성합니다.
3. PR 제목, 요약(변경 이유/내용/영향), 테스트 계획을 자동 생성합니다.
4. 관련 이슈를 자동 링크합니다 (`Closes #123`).

### 4.4 코드리뷰 — 교차 검증의 핵심

코드리뷰는 이 워크플로우의 핵심 단계입니다. 코드를 작성한 AI(Claude Code)와 **다른 AI(Gemini CLI)**가 리뷰해야 교차 검증의 의미가 있습니다.

## 5. ai-review.sh 동작 구조

`scripts/ai-review.sh`는 멀티 AI 리뷰 파이프라인의 진입점입니다. 2단계 구조로 동작합니다.

### 5.1 1단계: Gemini CLI 리뷰

Gemini CLI가 PR diff를 분석하여 두 가지 관점에서 리뷰를 수행하고, `gh` 명령어로 PR에 직접 코멘트를 게시합니다.

```bash
# 사용법
./scripts/ai-review.sh <PR번호>
```

**아키텍처/설계 리뷰** (`review-prompt-architecture.md`):
- 계층 간 의존성 방향 위반 (Layer leak)
- 도메인 모델링 안티패턴 (Anemic domain, Aggregate bypass 등)
- 의존성 조립 안티패턴 (Global singleton, Manual wiring 등)
- 설계 적합성과 잘한 점

**코드 품질/운영 리뷰** (`review-prompt-quality.md`):
- FastAPI/API 안티패턴 (Untyped response, Domain HTTP leak 등)
- DB/영속성 안티패턴 (N+1 query, Transaction ambiguity 등)
- 외부 연동 안티패턴 (No resilience, Half state 등)
- 테스트 품질과 운영 안정성

두 리뷰 프롬프트 모두 프로젝트 아키텍처(Hexagonal + DDD, 5계층)를 명시적으로 주입하여, AI 리뷰어가 프로젝트 맥락을 정확히 이해한 상태에서 리뷰합니다. 출력은 Critical / Warning / Good / Watchlist 4단계로 구조화됩니다.

### 5.2 2단계: Claude Code 종합 판단

Claude Code가 Gemini의 리뷰 코멘트를 읽고, 각 지적에 대해 **동의/반박/보류**를 판단합니다.

```bash
# ai-review.sh 내부에서 자동 실행
# 또는 Claude Code 스킬로 직접 호출
/review <PR번호>
```

판단 기준:
- 현재 프로젝트 아키텍처(Hexagonal + DDD)에 실제로 맞는 지적인지
- MVP 단계에서 지금 고쳐야 할 문제인지, 나중에 해도 되는 문제인지
- 안티패턴 태그가 정확한지 (잘못된 분류는 정정)
- 지적이 과도하지 않은지 (불필요한 추상화 요구 등)

판단 결과는 PR에 종합 코멘트로 게시됩니다.

```
### ✅ 동의
- 코멘트 요약 | 이유 | 수정 방향

### ❌ 반박
- 코멘트 요약 | 반박 근거

### ⏸️ 보류
- 코멘트 요약 | 추가 논의 필요 이유
```

### 5.3 Claude fallback 경로

Gemini CLI가 설치되어 있지 않은 환경에서는 Claude Code가 1단계 리뷰도 대행합니다.

```bash
if command -v gemini &>/dev/null; then
    # Gemini 1단계 → Claude 2단계 (정상 경로)
else
    # Claude Code가 단독으로 리뷰 (fallback)
fi
```

fallback 경로에서는 Claude Code가 리뷰 결과를 JSON 배열로 출력하고, `gh api`로 파일별 라인 코멘트를 직접 게시합니다. 교차 검증의 이점은 사라지지만, 리뷰 자동화 자체는 유지됩니다.

### 5.4 전체 흐름 요약

```
ai-review.sh <PR번호>
├── [Gemini 있음] 정상 경로
│   ├── Gemini → 아키텍처/설계 리뷰 → PR 코멘트
│   ├── Gemini → 코드 품질/운영 리뷰 → PR 코멘트
│   └── Claude → Gemini 코멘트 읽기 → 동의/반박/보류 판단 → PR 코멘트
└── [Gemini 없음] fallback 경로
    └── Claude → 아키텍처 + 품질 리뷰 → JSON → 라인별 코멘트 게시
```

## 6. 장점 및 이점

**구조적 품질 보증**: 코드를 작성한 AI와 다른 AI가 리뷰하므로, 단일 모델의 맹점에서 비롯되는 오류를 사전에 차단합니다. 실제로 Gemini가 발견한 환각(존재하지 않는 오타를 지적)을 Claude가 반박하여 걸러낸 사례가 확인되었습니다.

**검토 부담 경감**: AI 간 교차 검증을 거친 결과물만 사람에게 도달하므로, 사람의 리뷰 범위가 "전체 코드"에서 "AI 간 의견이 갈리는 지점"으로 좁아집니다.

**비용 효율**: Gemini CLI 무료 티어(일일 1,000회)를 1단계 리뷰에 배치하여, Claude Code의 유료 호출을 최종 판단에만 집중합니다.

**프로세스 표준화**: CLAUDE.md로 프로젝트 규칙을 자동 주입하고, 리뷰 프롬프트에 아키텍처 기준을 명시하므로, 세션이 바뀌어도 동일한 품질 기준이 유지됩니다.

**높은 처리량**: 1인 개발자가 하루에 10개의 PR을 생성·리뷰·머지할 수 있는 속도를 실현합니다.

### 7.3 워크플로우 패턴

실제 운용에서 드러난 패턴은 다음과 같습니다.

1. **인프라 PR은 빠르게 머지**: lint, hook, Makefile 등 설정 변경은 리뷰 없이 즉시 머지합니다.
2. **기능 PR은 교차 검증 후 머지**: 비즈니스 로직이 포함된 PR은 Gemini + Claude 2단계 리뷰를 거칩니다.
3. **리뷰 지적은 즉시 반영 또는 이슈 분리**: 동의하는 지적은 같은 PR에서 수정하고, 범위 밖 지적은 별도 이슈로 추적합니다.

## 8. 프로젝트 세팅 구조

이 워크플로우를 지원하기 위해 프로젝트에 구성해야 할 파일 구조입니다.

```
project-root/
├── CLAUDE.md                    # 핵심 규칙 인라인 + @import
├── .claude/
│   ├── rules/
```

### 8.1 리뷰 프롬프트 설계 원칙

리뷰 프롬프트에는 다음 요소를 반드시 포함해야 합니다.

| 요소 | 이유 |
|------|------|
| 프로젝트 아키텍처 계층 정의 | AI가 계층 간 의존성 위반을 정확히 판단 |
| 구체적인 안티패턴 목록 | 추상적 지적 방지, 분류 태그 일관성 확보 |
| 출력 형식 (Critical/Warning/Good/Watchlist) | 심각도별 분류로 우선순위 판단 용이 |
| "잘한 점도 반드시 언급" 지시 | 문제 지적만 하면 과도한 리뷰가 되므로 균형 유지 |
| 리뷰어 태그 | 어떤 AI가 남긴 코멘트인지 추적 가능 |

## 9. 한계점 및 고려사항

**모델 간 환각 불일치**: Gemini가 존재하지 않는 오타를 지적하거나, 잘못된 라인 번호를 참조하는 사례가 확인되었습니다. 2단계 판단이 없으면 이런 오탐이 그대로 반영될 위험이 있습니다. 교차 검증 구조가 이 문제를 완화하지만, 완전히 제거하지는 못합니다.

**비용**: Claude Code의 API 호출 비용이 주요 비용 요인입니다. Gemini CLI 무료 티어를 1단계에 배치하여 비용을 억제하지만, 대규모 diff에서는 Claude 2단계 판단의 토큰 소비가 증가합니다.

**429 레이트 리밋**: Gemini CLI 무료 티어는 분당 60회, 일일 1,000회 제한이 있습니다. PR이 대량으로 생성되는 시점에 429 에러가 발생할 수 있으며, 리뷰 프롬프트가 두 번 호출(아키텍처 + 품질)되므로 실질적으로 PR 500개가 일일 한계입니다.

**도구 간 컨텍스트 전달**: 세 CLI 도구 간에 컨텍스트를 직접 공유하는 공식 메커니즘이 없습니다. `ai-review.sh`가 파일 기반(마크다운 리포트 출력 → `gh api`로 PR에 게시 → 다른 도구가 읽기)으로 전달하며, 이 과정에서 정보 손실이 발생할 수 있습니다.

**리뷰 품질의 프롬프트 의존**: 리뷰 프롬프트에 프로젝트 아키텍처를 명시하지 않으면, AI 리뷰어가 일반적인 기준으로 리뷰하여 프로젝트 맥락과 맞지 않는 지적을 합니다. 프로젝트 구조가 변경되면 리뷰 프롬프트도 함께 업데이트해야 합니다.

**Codex CLI 미검증**: 현재 Claude Code + Gemini CLI 2도구 체계만 실전에서 검증되었습니다. Codex CLI를 테스트 병렬 생성이나 대안 구현에 도입하는 것은 향후 과제입니다.
