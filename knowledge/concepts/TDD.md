# TDD

> 테스트를 먼저 작성하고, 그 테스트를 통과하는 최소한의 코드를 구현한 뒤, 리팩터링하는 개발 방법론입니다.

---

## 1. 정의

TDD(Test-Driven Development)는 Kent Beck이 1990년대 후반 익스트림 프로그래밍(XP)의 일부로 정립한 개발 방법론입니다. 핵심 원칙은 단순합니다: **프로덕션 코드를 작성하기 전에 반드시 실패하는 테스트를 먼저 작성합니다.**

일반적인 개발 흐름이 "코드 작성 → 테스트 작성"인 반면, TDD는 "테스트 작성 → 코드 작성 → 리팩터링"의 순서를 따릅니다. 이 순서의 전환이 설계 품질과 결함 밀도에 근본적인 차이를 만듭니다.

---

## 2. 등장 배경 및 필요성

TDD는 소프트웨어 개발에서 반복되는 세 가지 문제를 해결하기 위해 등장했습니다.

- **후행 테스트의 구조적 한계**: 코드를 먼저 작성하면 테스트하기 어려운 구조가 만들어지기 쉽습니다. 강한 결합, 숨겨진 의존성, 부수 효과가 있는 함수 등이 대표적입니다. 테스트를 나중에 추가하려 할 때 이미 구조가 굳어져 테스트 작성 비용이 급증합니다.
- **요구사항과 구현의 괴리**: 명세를 코드로 번역하는 과정에서 해석 오류가 발생합니다. TDD는 요구사항을 테스트 형태로 먼저 명시함으로써 "무엇을 만들어야 하는지"를 코드 수준에서 확정합니다.
- **회귀 결함의 누적**: 기능 추가나 리팩터링 시 기존 기능이 깨지는 회귀 결함은 프로젝트 규모에 비례하여 증가합니다. TDD로 축적된 테스트 스위트는 변경의 안전망 역할을 합니다.

---

## 3. 작동 원리 / 핵심 개념

### 3.1 레드-그린-리팩터 사이클

TDD의 핵심 루프는 세 단계로 구성됩니다.

```
┌──────────┐     ┌──────────┐     ┌──────────────┐
│  🔴 레드  │ ──▶ │  🟢 그린  │ ──▶ │  🔵 리팩터링  │
│ 실패하는   │     │ 통과하는   │     │  구조 개선     │
│ 테스트 작성 │     │ 최소 코드  │     │  테스트 유지   │
└──────────┘     └──────────┘     └──────┬───────┘
      ▲                                  │
      └──────────────────────────────────┘
```

**레드 단계**: 아직 구현되지 않은 기능에 대한 테스트를 작성합니다. 이 단계에서 테스트는 반드시 실패해야 합니다. 실패를 확인하는 것은 테스트 자체가 올바르게 작동하는지 검증하는 과정입니다. 이 단계에서 모듈, 클래스, 함수의 인터페이스를 설계하게 됩니다.

**그린 단계**: 테스트를 통과하는 가장 단순한 코드를 작성합니다. 하드코딩도 허용됩니다. 목표는 속도이며, 우아함은 다음 단계의 몫입니다. 테스트가 요구하지 않는 코드를 추가하지 않습니다.

**리팩터링 단계**: 모든 테스트가 통과하는 상태에서 코드 구조를 개선합니다. 중복 제거, 네이밍 개선, 책임 분리 등을 수행합니다. 리팩터링 후에도 모든 테스트가 통과해야 합니다. 이 단계를 생략하면 코드가 점진적으로 정리되지 않은 조각들의 집합이 됩니다.

### 3.2 테스트 피라미드

Mike Cohn이 제안한 테스트 피라미드는 테스트를 세 계층으로 분류합니다.

```
        /  E2E  \          ← 느림, 비쌈, 소수
       /----------\
      / 통합 테스트  \       ← 중간
     /----------------\
    /   단위 테스트     \    ← 빠름, 저렴, 다수
   /____________________\
```

**단위 테스트**: 함수나 메서드 단위의 동작을 검증합니다. 외부 의존성을 테스트 더블로 대체하여 격리합니다. 밀리초 단위로 실행되며, 전체 테스트의 70~80%를 차지해야 합니다.

**통합 테스트**: 여러 컴포넌트가 협력하는 동작을 검증합니다. 데이터베이스 연결, API 호출 등 실제 의존성을 포함합니다. 단위 테스트보다 느리지만 컴포넌트 간 상호작용 결함을 발견합니다. 전체의 15~20%가 적절합니다.

**E2E 테스트**: 사용자 관점에서 전체 시스템을 검증합니다. 브라우저 자동화(Cypress, Playwright 등)를 사용하여 실제 사용 시나리오를 재현합니다. 가장 느리고 깨지기 쉬우므로 핵심 시나리오에만 제한하여 전체의 5~10%를 유지합니다.

### 3.3 테스트 더블

Gerard Meszaros가 정의한 테스트 더블(Test Double)은 프로덕션 객체를 대체하는 테스트용 객체의 총칭입니다. 영화의 스턴트 더블에서 이름을 따왔습니다.

| 유형 | 역할 | 검증 방식 | 예시 |
|------|------|-----------|------|
| 더미(Dummy) | 매개변수를 채우기 위해 전달만 되고 실제로 사용되지 않음 | 없음 | 빈 객체, null |
| 스텁(Stub) | 미리 정해진 응답을 반환 | 상태 검증 | 항상 `true`를 반환하는 인증 서비스 |
| 페이크(Fake) | 동작하는 구현체이나 프로덕션에는 부적합 | 상태 검증 | 인메모리 데이터베이스 |
| 스파이(Spy) | 스텁 + 호출 기록을 남김 | 상태/행위 검증 | 발송된 이메일 수를 기록하는 메일 서비스 |
| 목(Mock) | 기대하는 호출을 사전 정의하고, 그대로 호출되었는지 검증 | 행위 검증 | `send()`가 정확히 1회 호출되었는지 검증 |

Martin Fowler는 스텁과 목의 핵심 차이를 **상태 검증 대 행위 검증**으로 설명합니다. 스텁은 "결과가 올바른가?"를 확인하고, 목은 "올바른 방식으로 호출되었는가?"를 확인합니다.

### 3.4 AAA 패턴

테스트 코드의 가독성을 높이는 구조화 패턴입니다.

```python
def test_apply_discount_reduces_total():
    # Arrange — 사전 조건 설정
    cart = ShoppingCart()
    cart.add_item(Item("신발", 100_000))

    # Act — 테스트 대상 동작 실행
    cart.apply_discount(percent=10)

    # Assert — 기대 결과 검증
    assert cart.total == 90_000
```

- **Arrange(준비)**: 테스트에 필요한 객체와 데이터를 설정합니다.
- **Act(실행)**: 테스트 대상 동작을 한 번 호출합니다.
- **Assert(검증)**: 기대한 결과가 나왔는지 확인합니다.

## 4. 장점 및 이점

- **설계 품질 향상**: 테스트를 먼저 작성하면 자연스럽게 느슨한 결합과 명확한 인터페이스를 가진 설계가 도출됩니다. 테스트하기 어려운 코드는 대체로 설계가 나쁜 코드입니다.
- **결함 밀도 감소**: IBM과 Microsoft의 사례 연구에서 TDD 적용 시 출시 전 결함이 최대 40~90% 감소한 것으로 보고되었습니다.
- **리팩터링 안전망**: 포괄적인 테스트 스위트가 있으면 구조 변경 시 기존 동작이 유지되는지 즉시 확인할 수 있습니다. 리팩터링에 대한 심리적 부담이 크게 줄어듭니다.
- **살아있는 문서**: 테스트 코드는 "이 모듈이 무엇을 해야 하는가"를 실행 가능한 형태로 기록합니다. 주석이나 문서보다 정확하고, 코드와 항상 동기화됩니다.
- **디버깅 시간 단축**: 결함이 발생하면 직전에 작성한 소량의 코드 내에서 원인을 찾으면 되므로 디버깅 범위가 좁습니다.

---

## 5. 한계점 및 고려사항

- **초기 개발 속도 저하**: 연구에 따르면 TDD는 초기 개발 시간을 15~35% 증가시킵니다. 유지보수 단계에서 이 비용을 회수하지만, 프로토타이핑이나 탐색적 개발에는 부적합할 수 있습니다.
- **테스트 유지보수 비용**: 프로덕션 코드가 변경되면 관련 테스트도 함께 수정해야 합니다. 구현 세부사항에 지나치게 결합된 테스트는 리팩터링을 오히려 방해합니다.
- **목 과다 사용 문제**: 행위 검증에 과도하게 의존하면 테스트가 구현에 강하게 결합되어, 내부 구조를 바꿀 때마다 테스트가 깨집니다. 가능하면 상태 검증을 우선하고, 부수 효과를 검증할 때만 목을 사용합니다.
- **학습 곡선**: 적절한 테스트 크기 설정, 테스트 순서 결정, 테스트 더블의 올바른 사용법 등은 숙달에 시간이 필요합니다.
- **UI/비결정적 로직의 어려움**: 시각적 레이아웃, 난수 기반 로직, 외부 API 의존 코드 등은 TDD로 접근하기 어려운 영역입니다.

## 6. 실무 적용 가이드

### 6.1 TDD가 효과적인 영역

| 영역 | 이유 |
|------|------|
| 비즈니스 로직 | 명확한 입출력이 있고 규칙이 복잡할수록 테스트의 가치가 높음 |
| 유틸리티/헬퍼 함수 | 순수 함수는 TDD에 가장 적합한 대상 |
| API 엔드포인트 | 요청-응답 계약이 명확하여 테스트 정의가 쉬움 |
| 버그 수정 | 버그를 재현하는 테스트를 먼저 작성하면 재발을 방지할 수 있음 |

### 6.2 TDD가 비효율적인 영역

| 영역 | 이유 |
|------|------|
| UI 레이아웃 | 픽셀 단위 검증은 비용 대비 효과가 낮음 |
| 프로토타이핑 | 요구사항이 불확실한 탐색 단계에서는 테스트가 발목을 잡음 |
| 외부 API 통합 | 제어 불가능한 외부 시스템은 계약 테스트가 더 적합 |
| 설정/글루 코드 | 단순 연결 코드는 통합 테스트로 충분 |

### 6.3 실무 지침

- **테스트 이름에 의도를 담습니다**: `test_1`이 아니라 `test_expired_coupon_returns_zero_discount`처럼 작성합니다.
- **테스트당 하나의 동작만 검증합니다**: 여러 동작을 한 테스트에 넣으면 실패 원인 파악이 어렵습니다.
- **100% 커버리지를 목표로 하지 않습니다**: 의미 있는 커버리지와 변이 테스트(Mutation Testing) 점수를 함께 추적하는 것이 더 효과적입니다.
- **테스트 코드와 프로덕션 코드를 별도로 리팩터링합니다**: 둘을 동시에 바꾸면 무엇이 문제를 일으켰는지 알 수 없습니다.
- **작은 커밋을 자주 합니다**: 레드-그린-리팩터 한 사이클마다 커밋하면 잘못된 방향으로 갔을 때 되돌리기 쉽습니다.

### 6.4 주요 테스트 프레임워크

| 언어 | 프레임워크 | 특징 |
|------|-----------|------|
| 파이썬 | pytest | 간결한 문법, 풍부한 플러그인 생태계, 픽스처 시스템 |
| 자바스크립트 | Jest, Vitest | 스냅샷 테스트, 모킹 내장, 빠른 실행 |
| 자바 | JUnit 5, Mockito | 어노테이션 기반, 목 프레임워크 표준 |
| Go | testing (표준 라이브러리) | 외부 의존성 없이 테스트 작성 가능 |
