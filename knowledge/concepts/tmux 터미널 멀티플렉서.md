# tmux 터미널 멀티플렉서

> 단일 터미널 창에서 여러 개의 가상 터미널 세션을 생성하고 관리하는 서버-클라이언트 기반 터미널 멀티플렉서입니다.

---

## 1. 정의

tmux(Terminal Multiplexer)는 하나의 터미널 창 안에서 여러 개의 독립적인 터미널 세션을 생성, 분할, 전환할 수 있는 도구입니다.

터미널 에뮬레이터와 혼동하기 쉬우나, 역할이 명확히 다릅니다.

| 구분 | 터미널 에뮬레이터 | 터미널 멀티플렉서 |
|------|-------------------|-------------------|
| 역할 | 물리 터미널을 소프트웨어로 구현 | 하나의 터미널 안에서 여러 세션 관리 |
| 예시 | [[Ghostty 단축키\|Ghostty]], iTerm2, Alacritty | tmux, GNU Screen |
| 계층 | OS 위에서 동작하는 GUI 애플리케이션 | 터미널 에뮬레이터 안에서 동작하는 CLI 도구 |
| 세션 영속성 | 창을 닫으면 세션 소멸 | 클라이언트를 종료해도 세션 유지 |

터미널 에뮬레이터는 "어떤 창에서 터미널을 띄울 것인가"를 결정하고, 멀티플렉서는 "그 창 안에서 작업을 어떻게 배치할 것인가"를 결정합니다.

---

## 2. 등장 배경 및 필요성

- **원격 서버 작업의 불안정성**: SSH 연결이 끊기면 실행 중이던 프로세스가 함께 종료됩니다. 장시간 배포 작업이나 데이터 마이그레이션 중 네트워크 단절은 치명적인 문제로 이어집니다.
- **단일 터미널 창의 한계**: 코드 편집, 로그 모니터링, 데이터베이스 쿼리를 동시에 수행하려면 여러 터미널 창을 열어야 하며, 창 전환 비용이 큽니다.
- **반복적인 환경 구성**: 프로젝트마다 동일한 터미널 레이아웃을 수동으로 구성하는 것은 비효율적입니다.
- **다중 프로세스 병렬 제어**: 여러 AI 에이전트나 빌드 프로세스를 동시에 실행하고 모니터링해야 하는 현대 개발 환경에서, 단일 터미널로는 한계가 있습니다.

---

## 3. 작동 원리 / 핵심 개념

### 3.1 서버-클라이언트 구조

tmux는 백그라운드에서 실행되는 **서버 프로세스**와 사용자가 상호작용하는 **클라이언트 프로세스**로 구성됩니다.

```
┌─────────────────────────────────────────┐
│              tmux 서버 (백그라운드)         │
│  ┌──────────┐  ┌──────────┐             │
│  │  세션 A   │  │  세션 B   │  ...       │
│  └──────────┘  └──────────┘             │
└─────────────────────────────────────────┘
        ↑               ↑
   클라이언트 1      클라이언트 2
   (터미널 창)      (터미널 창)
```

클라이언트가 종료되어도 서버는 살아 있으므로, 나중에 다시 연결(attach)하면 이전 작업 상태가 그대로 유지됩니다.

### 3.2 세션 / 윈도우 / 패인 계층 구조

tmux는 세 단계의 계층으로 작업 공간을 구성합니다.

```
세션 (Session)
 └── 윈도우 (Window) — 탭과 유사
      └── 패인 (Pane) — 분할된 영역
```

| 계층 | 개념 | 비유 |
|------|------|------|
| 세션 | 독립적인 작업 환경 단위 | 프로젝트별 워크스페이스 |
| 윈도우 | 세션 안의 전체 화면 단위 | 브라우저 탭 |
| 패인 | 윈도우를 분할한 개별 터미널 | IDE의 분할 편집기 |

### 3.3 접두사 키 (Prefix Key)

tmux의 모든 단축키는 **접두사 키**를 먼저 누른 뒤 명령 키를 입력하는 2단계 방식으로 동작합니다. 기본 접두사 키는 `Ctrl+B`입니다.

예를 들어 윈도우를 새로 만들려면, `Ctrl+B`를 누른 뒤 손을 떼고 `c`를 누릅니다.

---

## 4. 핵심 단축키

모든 단축키는 기본 접두사 키 `Ctrl+B`를 먼저 입력한 뒤 사용합니다.

### 4.1 세션 관리

| 단축키 | 동작 |
|--------|------|
| `d` | 현재 세션에서 분리(detach) |
| `s` | 세션 목록 표시 및 전환 |
| `$` | 현재 세션 이름 변경 |
| `(` | 이전 세션으로 전환 |
| `)` | 다음 세션으로 전환 |

**명령줄에서 사용하는 세션 명령:**

| 명령 | 동작 |
|------|------|
| `tmux new -s 이름` | 새 세션 생성 (이름 지정) |
| `tmux ls` | 세션 목록 확인 |
| `tmux attach -t 이름` | 기존 세션에 연결 |
| `tmux kill-session -t 이름` | 세션 종료 |

### 4.2 윈도우 관리

| 단축키 | 동작 |
|--------|------|
| `c` | 새 윈도우 생성 |
| `,` | 현재 윈도우 이름 변경 |
| `w` | 윈도우 목록 표시 및 전환 |
| `n` | 다음 윈도우로 이동 |
| `p` | 이전 윈도우로 이동 |
| `0`~`9` | 해당 번호 윈도우로 이동 |
| `&` | 현재 윈도우 종료 (확인 요청) |

### 4.3 패인 관리

| 단축키 | 동작 |
|--------|------|
| `%` | 세로 분할 (좌우) |
| `"` | 가로 분할 (상하) |
| `방향키` | 해당 방향 패인으로 포커스 이동 |
| `o` | 다음 패인으로 포커스 이동 |
| `x` | 현재 패인 종료 (확인 요청) |
| `z` | 현재 패인 확대/복원 토글 |
| `{` | 현재 패인을 이전 위치로 이동 |
| `}` | 현재 패인을 다음 위치로 이동 |
| `Ctrl+방향키` | 패인 크기 조정 |
| `Space` | 패인 레이아웃 순환 변경 |
| `!` | 현재 패인을 독립 윈도우로 분리 |

### 4.4 기타

| 단축키 | 동작 |
|--------|------|
| `?` | 단축키 목록 표시 |
| `:` | 명령 모드 진입 |
| `[` | 복사 모드 진입 (스크롤, 텍스트 선택) |
| `]` | 붙여넣기 |
| `t` | 시계 표시 |

## 5. 장점 및 이점

- **세션 영속성**: SSH 연결이 끊기거나 터미널을 닫아도 서버 프로세스가 세션을 유지합니다. `tmux attach`로 언제든 복구할 수 있어, 원격 서버에서의 장시간 작업이 안전합니다.
- **작업 공간 분할**: 하나의 화면에서 코드 편집기, 로그 출력, 데이터베이스 클라이언트를 동시에 배치할 수 있습니다. IDE 수준의 통합 작업 환경을 터미널만으로 구현합니다.
- **스크립팅 자동화**: 세션 생성, 윈도우 분할, 명령 실행을 쉘 스크립트로 자동화할 수 있습니다. 프로젝트별 개발 환경을 한 번의 명령으로 즉시 구성합니다.
- **AI 에이전트 병렬 제어**: 여러 AI 모델이나 에이전트를 각각의 패인에서 동시에 실행하고, 한 화면에서 전체 파이프라인을 모니터링할 수 있습니다.
- **다중 클라이언트 접속**: 하나의 세션에 여러 클라이언트가 동시에 연결할 수 있어, 페어 프로그래밍이나 실시간 모니터링 공유에 유용합니다.

---

## 6. 한계점 및 고려사항

- **학습 곡선**: 접두사 키 기반의 2단계 단축키 체계가 직관적이지 않으며, 세션/윈도우/패인 개념을 익히는 데 시간이 필요합니다.
- **마우스 지원 제한**: 기본 설정에서는 마우스를 사용할 수 없습니다. `set -g mouse on` 설정으로 활성화할 수 있으나, 터미널 에뮬레이터의 네이티브 마우스 동작과 충돌할 수 있습니다.
- **클립보드 통합 복잡성**: tmux 내부의 복사 버퍼와 시스템 클립보드는 기본적으로 분리되어 있습니다. OS별로 별도 설정이나 플러그인(tmux-yank 등)이 필요합니다.
- **터미널 색상/렌더링 이슈**: tmux가 중간 계층으로 동작하면서 256색이나 트루컬러 출력에 문제가 발생할 수 있습니다. `default-terminal` 설정을 올바르게 지정해야 합니다.

### tmux vs GNU Screen 비교

| 항목 | tmux | GNU Screen |
|------|------|------------|
| 패인 분할 | 수평/수직 자유 분할, 직관적 | 기본적으로 수평만 지원, 수직 분할은 패치 필요 |
| 스크립팅 | 명령줄 인터페이스 체계적, 자동화 용이 | 가능하나 인터페이스가 비교적 비체계적 |
| 상태 표시줄 | 기본 제공, 커스터마이징 용이 | 수동 설정 필요 |
| 플러그인 생태계 | tpm 기반 활발한 플러그인 생태계 | 제한적 |
| 직렬 포트 연결 | 미지원 | 기본 지원 |
| 기본 설치 여부 | 별도 설치 필요 | 다수 리눅스 배포판에 기본 포함 |
| 커뮤니티 활성도 | 활발 (GitHub 기반 개발) | 유지보수 수준 |
| 학습 난이도 | 기능이 많아 초기 학습량이 큼 | 기능이 단순하여 빠르게 익힘 |

현대 개발 환경에서는 패인 관리, 스크립팅 자동화, 플러그인 확장성 측면에서 tmux가 우위에 있으며, 대부분의 경우 tmux를 선택하는 것이 권장됩니다.

## 7. 실무 적용 가이드

### 7.1 추천 .tmux.conf 설정

```bash
# ──────────────────────────────────────
# 기본 설정
# ──────────────────────────────────────

# 접두사 키를 Ctrl+A로 변경 (선택)
# set -g prefix C-a
# unbind C-b
# bind C-a send-prefix

# ESC 지연 제거 (Vim 사용자 필수)
set -sg escape-time 0

# 스크롤백 버퍼 확장
set -g history-limit 50000

# 트루컬러 지원
set -g default-terminal "tmux-256color"
set -ga terminal-overrides ",*256col*:Tc"

# 마우스 지원 활성화
set -g mouse on

# ──────────────────────────────────────
# 윈도우 및 패인 번호
# ──────────────────────────────────────

# 윈도우/패인 번호를 1부터 시작
set -g base-index 1
set -g pane-base-index 1

# 윈도우 삭제 시 번호 재정렬
set -g renumber-windows on

# ──────────────────────────────────────
# 패인 분할 단축키 개선
# ──────────────────────────────────────

# 현재 경로를 유지하며 분할
bind | split-window -h -c "#{pane_current_path}"
bind - split-window -v -c "#{pane_current_path}"
unbind '"'
unbind %

# ──────────────────────────────────────
# 패인 이동 (Alt+방향키, 접두사 키 불필요)
# ──────────────────────────────────────

bind -n M-Left select-pane -L
bind -n M-Right select-pane -R
bind -n M-Up select-pane -U
bind -n M-Down select-pane -D

# ──────────────────────────────────────
# 설정 즉시 반영
# ──────────────────────────────────────

bind r source-file ~/.tmux.conf \; display "설정을 다시 불러왔습니다."
```

### 7.2 프로젝트 환경 자동화 스크립트

쉘 스크립트로 프로젝트별 tmux 환경을 자동 구성할 수 있습니다.

```bash
#!/bin/bash
# dev-session.sh — 개발 환경 자동 구성

SESSION="my-project"

# 기존 세션이 있으면 연결, 없으면 생성
tmux has-session -t $SESSION 2>/dev/null

if [ $? != 0 ]; then
    # 세션 생성 + 첫 번째 윈도우: 코드 편집기
    tmux new-session -d -s $SESSION -n "editor"
    tmux send-keys -t $SESSION:editor "cd ~/projects/my-project && vim ." Enter

    # 두 번째 윈도우: 서버 + 로그
    tmux new-window -t $SESSION -n "server"
    tmux split-window -h -t $SESSION:server
    tmux send-keys -t $SESSION:server.1 "cd ~/projects/my-project && make run" Enter
    tmux send-keys -t $SESSION:server.2 "tail -f ~/projects/my-project/logs/app.log" Enter

    # 세 번째 윈도우: 데이터베이스
    tmux new-window -t $SESSION -n "db"
    tmux send-keys -t $SESSION:db "psql -d mydb" Enter

    # 첫 번째 윈도우로 포커스
    tmux select-window -t $SESSION:editor
fi

tmux attach -t $SESSION
```

### 7.3 AI 에이전트 병렬 실행 패턴

여러 AI 에이전트를 동시에 실행하고 모니터링하는 구성입니다.

```bash
#!/bin/bash
# ai-agents.sh — 다중 AI 에이전트 병렬 실행

SESSION="ai-pipeline"

tmux new-session -d -s $SESSION -n "agents"

# 4분할 레이아웃 구성
tmux split-window -h -t $SESSION:agents
tmux split-window -v -t $SESSION:agents.1
tmux split-window -v -t $SESSION:agents.2

# 각 패인에 에이전트 할당
tmux send-keys -t $SESSION:agents.1 "claude --session code-review" Enter
tmux send-keys -t $SESSION:agents.2 "claude --session test-gen" Enter
tmux send-keys -t $SESSION:agents.3 "claude --session doc-writer" Enter
tmux send-keys -t $SESSION:agents.4 "echo '=== 모니터링 패인 ===' && watch -n 5 'ls -la output/'" Enter

# 모든 패인 크기 균등화
tmux select-layout -t $SESSION:agents tiled

tmux attach -t $SESSION
```

이 패턴을 활용하면 [[멀티 AI 에이전트 개발 자동화 워크플로우]]에서 설계한 교차 검증 파이프라인을 실제로 운용할 수 있습니다.
